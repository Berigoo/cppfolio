Secure connection di iot device

perangkat IoT (Internet of Things) tanpa adanya "koneksi" ke luar baik itu menerima atau-pun mengirim data, maka perangkat tersebut bukanlah IoT melainkan oT . (sry.)

"koneksi" disini bisa melalui berbagai protocol seperti HTTP/S, Websocket, MQTT, etc.

Protocols

agar "data" dapat dilakukan exchange maka perlu adanya aturan atau rule standardisasi, untuk membedakan mana "data" dan mana header dari data tersebut, header ini dapat berupa informasi dari "data" tersebut dan/atau berupa informasi yang "mendukung" data tersebut, hal inilah yang dinamakan Protocol, tanpa adanya Protocol maka client, server, publisher, subscriber, atau apapun itu, akan kesulitan untuk extracting/parsing "data" tersebut dan kemana "data" itu akan pergi. Masing-masing Protocol memiliki caranya masing-masing untuk bertukar "data".

HTTP/S

sederhananya protocol ini tipe-nya Request-Response, sebelum response ada, client harus melakukan request terlebih dahulu. Jadi protocol ini tidak cocok untuk kasus dimana server harus mengirim data terlebih dahulu ke client (ya.. meskipun client bisa melakukan poll ke server, but...). Dan perlu diperhatikan setiap request dari client bersifat stateless yang berarti bahwa request satu dengan yang lainnya itu independent.
contoh HTTP/S struktur sederhana:

POST /path/to/endpoint HTTP/1.1 
Host: www.example.com 
User-Agent: Mozilla/5.0 
Accept: application/json
Content-Type: application/json
Content-Length: 23

{
	"data": "ABC123"
}

Request-line

  • POST, merupakan HTTP/S metode, dapat berupa GET, POST, PUT, DELETE, etc.
  • /path/to/endpoint, adalah path dari resource yang diminta.
  • HTPP/1.1, adalah versi HTTP/S yang digunakan.

Request header

merupakan informasi / metadata dari request dan response. header berada antara Request-line dan body, berbentuk key-value pair. setiap key memiliki fungsi masing-masing terhadap request atau-pun response tersebut, seperti Host untuk target server hostname yang diminta oleh client, User-Agent informasi mengenai darimana request tersebut berasal, Accept untuk tipe response data yang diharapkan oleh client, dan Content-* ini berhubungan dengan body dari request tersebut. Untuk lebih lanjut bisa mampir ke sini.

Body

merupakan sebuah raw data yang dikirim dari/ke server. Section dari body dimulai dari blank line setelah Request Header section. {request-header-data}\r\n\r\n{body-data}. data ini bisa berupa apapun.

Umumnya developer jarang yang aware dengan konsep Protocol ini, karena banyak sekali fancy libraries / framework yang me-handle hal ini. Namun karena umumnya IoT Board itu terbatas dalam hal resources dan oleh sebab itu libraries yang dapat digunakan pun terbatas, dan terkadang hanya implementasi minimal-nya saja, jadi adakalanya perlu membuat atau-pun parsing HTTP/S message dari scratch.
Untuk lebih lanjut mengenai protocol ini bisa mampir ke web-nya mozilla.

WebSocket

Protocol ini merupakan solusi dari masalah HTTP/S sebelumnya. Protocol ini menggunakan single TCP untuk koneksi bidirectional. Protocol ini bagus untuk kasus yang sifatnya real-time. Pertama client akan melakukan opening handshake kemudian server me-response-nya, dan jika berhasil maka "gerbang" untuk data exchange terbuka, sekarang client dan server dapat bertukar data tanpa terikat konsep request-response, kemudian diakhiri dengan closing handshake yang di-initiate oleh salah satu peer (client atau server) dan peer yang lain me-respon-nya, lalu "gerbang" pun tertutup.
Untuk lebih lanjut bisa mampir ke sini.

MQTT

sekilas protocol ini mirip dengan WebSocket, hanya saja protocol komunikasi-nya ber-tipe pub-sub / broker-based. Protocol ini cocok untuk kasus dimana client lain tidak diketahui.

karena komunikasi antar client tidak bersifat direct (peer-to-peer), melainkan melalui server yang dinamakan Broker. client yang mengirim message (alias data) disebut publisher, dan yang menerima disebut subscriber. Sederhanya publisher mengirim message ke suatu topic, kemudian broker akan mengirim message tersebut ke client yang subscribe ke topic tersebut.

Connection

protocol hanyalah sebuah protocol perlu adanya suatu "koneksi" agar perangkat IoT dapat berkomunikasi dengan dunia luar. Dalam framework Arduino ada library WiFi.h, ESP8266WiFi.h untuk ESP8266 module. Contoh minimal-nya,

#include <ESP8266WiFi.h>

#define SSID "ssid"
#define PASS "pass"
#define PATH_NAME "/path"

void setup(){
	WiFi.mode(WIFI_STA); //set ke mode station
	WiFi.begin(SSID, PASS); //connect ke wifi
	
	while (WiFi.status() != WL_CONNECTED) { //tunggu hingga terkoneksi
	    delay(500);
	    Serial.print(".");
	}
}

void loop(){
	WiFiClient client; //variable untuk menyimpan client TCP connection
	
	if (client.connect("example.com", 80)) { //koneksi ke server
	    client.println("Hello"); //mengirimkan data "Hello" ke server
	}
}

dan koneksi ke server pun terbuat, hanya saja untuk mengirimkan HTTP request perlu merangkai-nya manual, contoh.

	if (client.connect("example.com", 80)) { //koneksi ke server
		//mengirim HTTP GET request ke server
		client.println("GET " + PATH_NAME + " HTTP/1.1"); 
		client.println("Host: example.com"; 
		client.println("Connection: close"); 
		client.println();

		while(client.connected()) {
			if (client.available()) {
				// consume response byte-per-byte dari server 
				char c = client.read();
			}
		}
	}

tentu ini akan akan sedikit eeh... tapi di Arduino framework ada beberapa minimal library untuk berbagai protocols agar mengurangi parsing secara manual, yaitu,

  • HTTP: HTTPClient.h, ESP8266HTTPClient.h.
  • WebSocket: WebSocketClient.h
  • MQTT: MqttClient.h
    maka dengan bantuan minimal library request sebelumnya dapat ditulis ulang jadi,
void loop(){
	WiFiClient client; //variable untuk menyimpan client TCP connection
	HTTPClient http;
	
	http.begin(client, "http://example.com:80" + PATH_NAME);
	int statusCode = http.GET(); //mengirim HTTP GET request
	
	if (statusCode == 200) {
		// get response body dari server
		String body = http.getString();
	}
}

Secure connection

request sebelumnya dikirim tanpa enkripsi, orang ketiga dapat melihatnya secara jelas. Untuk mengubahnya menjadi secure hanya ganti WiFiClient dengan WiFiClientSecure.

void loop(){
	WiFiClientSecure client; //variable untuk menyimpan client TCP connection
	HTTPClient http;

	client.setInsecure(); //lewwati server validasi

	http.begin(client, "https://example.com:443" + PATH_NAME);
	int statusCode = http.GET(); //mengirim HTTP GET request
	
	if (statusCode == 200) {
		// get response body dari server
		String body = http.getString();
	}
}

koneksi pun akan terenkripsi namun server tidak akan divalidasi, untuk kasus embedded system, setInsecure() sudah cukup tidak perlu adanya server validasi sepertinya, karena server-server yang akan dikunjungi oleh perangkat tersebut diketahui, berbeda dengan browser yang dimana perlu adanya certificate store untuk memvalidasi server-server yang dikunjungi pengguna, agar pengguna dapat terhindar dari server yang "jadi-jadian".

Sebenarnya kebanyakan IoT Boards dapat melakukan hal tersebut, untuk memvalidasi server perlu adanya Root CA yang dimana server certificate di sign olehnya atau oleh Intermediate CA darinya, maka,

const char* rootCA = "-----BEGIN CERTIFICATE-----\n{...}\n-----END CERTIFICATE-----\n";
void setup() { ... }
void loop(){
	WiFiClientSecure client; //variable untuk menyimpan client TCP connection
	HTTPClient http;

	client.setCACert(rootCA); //set CA certificate untuk koneksi sekarang
	
	http.begin(client, "https://example.com:443" + PATH_NAME);
	int statusCode = http.GET(); //mengirim HTTP GET request
	
	if (statusCode == 200) {
		// get response body dari server
		String body = http.getString();
	}
}

bisa juga menggunakan BearSSL::CertStore yang dimana lebih efektif jika ada puluhan CA, lumayan menghemat RAM. Untuk menggunakannya perlu adanya library yang me-handle Filesystem, contoh nya LittleFS, setelah terinstall maka,

  1. convert semua CA dari format .pem ke .der (binary)

      openssl -inform PEM -in {/path/ke/CA.crt.pem} -outform DER -out {/output/path/ca_xxx.der}

  2. archive semua CA ke dalam format .ar semisal bernama certs.ar

  3. flash filesystem ke perangkat IoT
    jika menggunakan PlatformIO tool maka pindahkan certs.ar ke dalam folder data di workdir, 

    workdir/

   certs.ar

 lib/
 src/
 test/
 platformio.ini

    selanjutnya tinggal jalankan,

    pio run -t uploadfs
